摘要

随着现代社会生活水平的提高，健康管理逐渐成为大众关注的焦点。然而，传统的饮食记录方式操作繁琐，导致用户难以长期坚持。本文设计并实现了一套健身饮食系统的设计与实现，旨在通过信息化手段简化健康管理流程。

系统前端采用 Vue 3 框架，结合 Pinia 状态管理和 Element Plus 组件库，构建了响应式的用户交互界面；后端基于 Node.js 和 Express 框架，实现了 RESTful API 接口。系统核心功能包括个性化能量代谢模型计算、AI 智能饮食分析、体重趋势可视化以及食谱管理等。

本文的研究亮点在于：第一，系统集成了 Mifflin-St Jeor 公式，根据用户体征动态生成每日热量及三大营养素目标；第二，引入了大语言模型（LLM）API，实现了通过自然语言描述自动解析食物营养数据的功能，解决了录入难的痛点；第三，通过 Canvas 技术在前端实现了图像质量压缩，有效提升了系统的运行效率。

经过功能测试和逻辑验证，本系统运行稳定，交互流程简洁，能够为用户提供科学、智能、便捷的健康管理方案。

关键词：健身饮食管理；Vue 3；Node.js；人工智能；数据可视化

健身饮食系统的设计与实现

第1章 绪论

1.1 研究背景
随着国民经济的快速发展 and 人民生活水平的显著提高，健康问题逐渐成为社会关注的焦点。现代生活节奏的加快以及饮食结构的多样化，导致肥胖、糖尿病及心血管疾病等健康问题日益普遍。科学的饮食管理和合理的健身计划是维持身体健康、控制体重的核心手段。然而，传统的饮食记录方式大多依赖于手工记录或繁琐的卡路里查询，用户往往因操作过于复杂而难以坚持。

与此同时，互联网技术的飞速发展，特别是人工智能（AI）和移动Web技术的普及，为健康管理提供了新的契机。通过大语言模型（LLM）的语义分析能力，可以极大地简化饮食数据的录入过程；结合科学的能量代谢计算模型（如 Mifflin-St Jeor 公式），可以为用户提供更加精准的个性化建议。因此，开发一套集智能化录入、科学化计算、可视化展示于一体的健身饮食系统的设计与实现具有重要的现实意义。

1.2 研究内容
本文主要研究并实现了一个健身饮食系统的设计与实现，其核心研究内容包括：
1. 个性化健康档案管理：研究如何根据用户的性别、年龄、身高、体重及活动强度，利用基础代谢率（BMR）和每日总能量消耗（TDEE）模型，动态生成个性化的每日热量和三大营养素目标。
2. 智能化饮食追踪与分析：探索将大语言模型集成到系统中，实现通过自然语言描述自动解析食物组成及热量数据的功能。
3. 数据可视化展示：研究如何利用图表技术直观呈现用户的热量摄入对比、营养占比以及体重变化趋势。
4. 前端图像处理优化：研究如何在客户端对用户上传的食物图片或头像进行实时压缩，以平衡用户体验与服务器存储压力。

1.3 研究目的
本研究旨在构建一个简单易用、功能完备且具有一定智能化的健康管理工具，具体目标如下：
1. 降低记录门槛：通过 AI 识别技术，让用户能以最自然的方式记录饮食，提高用户粘性。
2. 科学指导健身：提供基于人体代谢规律的数值参考，避免盲目节食或过度摄入。
3. 辅助决策支持：通过长期的体重和饮食数据积累，帮助用户识别饮食习惯中的不足，并给出针对性的改善建议。

1.4 论文组织

本论文共分为六个主要章节以及最后的结论与展望部分，具体章节安排及其研究内容如下：

1. 第1章 绪论：本章首先分析了现代社会背景下健康管理及健身饮食管理的重要性，指出了传统饮食记录方式存在的痛点。随后详细阐述了本文的研究内容、研究目的以及系统的设计初衷，最后介绍了论文的整体组织架构。

2. 第2章 相关技术：本章对系统开发过程中所涉及的核心技术栈进行了深入介绍。前端部分详细解析了 Vue 3 框架及其生态工具（如 Vite, Pinia, Element Plus, ECharts）的应用；后端部分阐述了 Node.js 环境下 Express 框架的架构设计，以及 JWT 鉴权、Bcrypt 加密和 OpenAI 大语言模型接口的集成方案。

3. 第3章 系统需求分析：本章从技术、社会、经济三个维度对项目进行了可行性论证。通过对普通用户和管理员两类角色的深度调研，明确了系统的功能性需求与非功能性需求，并利用用例图、用例规约以及类图等建模工具对系统逻辑进行了详细分析。

4. 第4章 系统设计：本章详细描述了系统的总体设计方案。包括基于前后端分离的系统架构设计，核心业务逻辑（如基于 Mifflin-St Jeor 公式的能量预算计算、AI 智能分析流）的设计，以及基于 JSON 存储模式的数据库表结构设计。

5. 第5章 系统实现：本章是论文的核心部分，重点展示了系统主要功能模块的界面实现与后端代码逻辑。详细说明了仪表盘数据可视化、AI 自然语言识别饮食记录、一键制订三餐计划以及前端图像压缩等特色功能的具体实现过程。

6. 第6章 软件测试：本章通过设计详细的测试用例对系统进行了全面的功能验证。包括针对核心计算公式的白盒测试，以及针对用户注册登录、饮食日志记录、AI 接口响应、管理员权限等关键环节的黑盒测试，并对测试结果进行了详细评估。

7. 结论：本章对健身饮食系统的设计与实现的全部研发工作进行了总结，阐述了系统的创新点与实际应用价值，并诚恳指出了当前版本在数据库扩展性等方面的不足，为未来的优化方向提出了展望。

第2章 相关技术

2.1 开发技术的介绍

2.1.1 网页端
本系统采用 B/S（Browser/Server）架构，网页端作为用户交互的主要入口，强调跨平台的兼容性和流畅的用户体验。通过响应式布局，确保系统在 PC 端和移动端浏览器上均能获得良好的显示效果。

2.1.2 程序前端
系统前端基于 Vue 3 框架构建，利用其优秀的响应式机制和组合式 API（Composition API）提升代码的可维护性。
1. Vite：作为下一代前端构建工具，提供了极速的热模块替换（HMR）和优化后的打包体验。
2. Pinia：用于系统的状态管理，负责存储用户信息、Token 以及每日饮食日志，确保数据在不同组件间的同步。
3. Element Plus：一套成熟的 UI 组件库，为系统提供了统一的视觉风格和丰富的交互组件。
4. ECharts：强大的可视化库，用于绘制体重曲线图和营养比例饼图。
5. Canvas API：在客户端实现图片压缩算法，在上传前将大尺寸图片转为低压缩率的 Base64 格式，减少网络传输负载。

2.1.3 程序后端
系统后端采用 Node.js 环境下的 Express 框架，构建 light、高效的 RESTful API 服务。
1. JWT (JSON Web Token)：实现基于令牌的身份验证机制，支持前后端分离的跨域访问并保障接口安全。
2. Bcryptjs：采用 Bcryptjs 对用户密码进行盐值哈希加密。这是一种基于 Blowfish 加密算法的不可逆加密技术，能够有效防止彩虹表攻击，保障用户账户的安全。
3. AI 接口技术：系统后端集成了 OpenAI SDK，作为标准化的通信协议接口。通过配置 OpenAI 兼容的 BaseURL，实现了与 Gemini 系列大语言模型的无缝对接。系统通过向该模型发送结构化的提示词（Prompt），实时获取非结构化饮食描述的语义分析结果，并将其转化为标准化的营养数据 JSON 对象。
4. Multer：处理 multipart/form-data 格式的图片上传请求。

2.2 开发工具的介绍

2.2.1 编辑器：Visual Studio Code
作为目前最流行的轻量级代码编辑器，VS Code 提供了丰富的插件生态（如 Volar, ESLint, Prettier），极大地提升了 Vue 项目的开发效率。

2.2.2 运行环境：Node.js
本系统的前端构建环境和后端服务器均运行在 Node.js 之上，实现了全栈 JavaScript 开发，降低了技术切换成本。

2.2.3 依赖管理工具：NPM/Vite
利用 NPM 进行第三方库的版本控制与管理，确保项目在不同环境下的一致性。Vite 负责将 modern JavaScript 代码编译、压缩为生产环境下可用的静态资源。

第3章 系统需求分析

3.1 可行性分析

3.1.1 技术可行性
系统前端采用 Vue 3 + Element Plus，能够快速构建出美观且交互流畅的界面。后端基于 Node.js 和 Express，能够处理并发请求并与 AI 接口无缝对接。数据持久化采用 JSON 文件存储，简化了数据库配置和维护成本。大语言模型接口的引入解决了复杂的语义识别问题。综上所述，系统在技术实现上是完全可行的。

3.1.2 社会可行性
随着“健康中国”战略的推进，公众对科学饮食和体重管理的需求日益增长。本系统旨在提供一个便捷、智能且免费的健康管理工具，符合当前社会的健康发展潮流，无法律或伦理风险，具有良好的社会推广前景。

3.1.3 经济可行性
系统采用开源的技术框架（Vue, Express, Element Plus），无额外商业授权费用。服务器端运行成本低，AI 接口调用按需付费且单次成本极低。对于毕业设计而言，开发与运行成本均在可控范围内，具有较高的经济性。

3.2 功能需求分析

3.2.1 用户分析
系统主要面向两类用户：
1. 普通用户：系统的核心受众，需求包括记录每日饮食、查看摄入热量、管理个人体重历史以及获取科学的饮食建议。
2. 管理员：负责系统的日常维护，需求包括监控用户活跃度、管理系统内置食谱库以及查看系统运行状态统计。

3.2.2 功能模块分析
根据实际项目功能，系统划分为以下核心模块：
1. 用户认证模块：提供用户注册、登录及 Token 鉴权功能。
2. 个人档案管理模块：支持用户录入性别、身高、体重、年龄、活动强度及健身目标（减脂、维持、增重），系统自动计算 BMR 和 TDEE。
3. 饮食追踪模块：支持手动录入食物数据（名称、热量、三大营养素）及 AI 智能录入（输入一段文字，AI 自动解析并计算数据）。
4. 数据可视化模块：通过仪表盘展示当日热量完成度，通过折线图展示近 7 日热量摄入对比，通过趋势图展示体重变化。
5. 食谱库模块：提供预设的健康食谱查询，支持管理员进行增加、修改和删除操作。

3.2.3 系统用例图设计思路
由于文档无法直接展示图像，本节详细描述用例图的设计逻辑：
参与者（Actor）：定义为“普通用户”和“管理员”。
普通用户用例：
1. 注册/登录：基础账号操作。
2. 设置个人档案：计算代谢水平的基础。
3. 记录饮食日志：核心业务流程，包含子用例“AI 智能分析”。
4. 查看统计报表：数据可视化展示。
5. 查询食谱：浏览健康饮食建议。
管理员用例：
1. 登录后台：管理员权限认证。
2. 系统统计查看：查看总用户数及食谱总数。
3. 管理食谱库：对公共食谱进行 CRUD（增删改查）操作。
4. 查看用户列表：监控注册用户信息。

3.2.4 用例规约
以“饮食追踪（AI 录入）”为例进行规约描述：
用例名称：AI 智能解析饮食记录。
基本流：
1. 用户进入饮食记录页面。
2. 用户在输入框输入一段自然语言描述（如“中午吃了一个三明治和一杯无糖咖啡”）。
3. 系统点击“分析”按钮，前端将请求发送至后端。
4. 后端调用大模型 API 进行语义解析，并返回结构化的营养素数据。
5. 前端展示解析结果，用户确认无误后点击保存。
异常流：AI 解析失败或网络超时，系统提示用户手动录入或稍后再试。

3.2.5 系统类图设计思路
系统基于轻量级架构，其核心类设计如下：
1. User 类：包含基本属性（username, role, avatar）及关联的 Profile 对象（包含 bmr, tdee 计算逻辑）。
2. DietLog 类：负责记录每日摄入数据，包含 date, meals (数组，包含每餐详情)。
3. Recipe 类：定义公共食谱结构，包含 name, ingredients, nutrition (热量、蛋白质等)。
4. AIAgent 类：作为工具类，负责封装与 AI 接口的通信逻辑及 JSON 结果解析。

第4章 系统设计

4.1 系统总体功能设计
本系统采用前后端分离的开发模式，总体架构由表示层（Vue 3 + Element Plus）、业务逻辑层（Pinia + Express）和数据持久层（JSON 数据库）组成。
1. 前端视图层：通过响应式路由 vue-router 管理页面跳转，包含登录页、仪表盘、饮食追踪器、食谱库、个人资料页及管理员后台。
2. 后端服务层：基于 Node.js 构建 RESTful API，处理用户认证、AI 数据分析、JSON 数据读写及静态资源管理。
3. 数据层：利用文件系统（fs 模块）实现对 users.json 和 recipes.json 的持久化操作，采用事务性写入确保数据的完整性。

4.2 主要功能设计

4.2.1 能量预算逻辑设计
系统的核心逻辑之一是基于用户体征计算每日热量缺口。设计逻辑如下：
1. BMR 计算：根据 Mifflin-St Jeor 公式，区分男女性别进行基础代谢计算。
2. TDEE 计算：将 BMR 乘以用户选定的活动强度系数（1.2 至 1.9）。
3. 目标热量动态调整：根据用户的 goal（减脂/增重/维持），在 TDEE 基础上进行 10%-20% 的加减。
4. 营养素配比设计：根据总热量目标，按照碳水 40-50%、蛋白质 30%、脂肪 20-30% 的比例自动生成三大营养素目标。

4.2.2 AI 智能分析流程设计
1. 请求发起：前端发送用户描述文字及当前用户的 Profile 数据。
2. 提示词构建：后端将文字嵌入预设的 Prompt 模板中，要求 AI 以严格的 JSON 格式输出包含 food, calories, carbs, protein, fat 字段的结果。
3. 结果解析与回传：后端对 AI 返回的字符串进行 JSON 校验和清洗，剔除无效的 Markdown 标签，确保前端能够稳定解析。

4.3 数据库设计

4.3.1 类图设计思路（架构层）
在设计层面，系统模型关系如下：
1. User (1) <-> (1) Profile：每个用户拥有唯一的个人体征档案。
2. User (1) <-> (N) WeightRecord：用户拥有多条按日期排序的体重记录，用于绘制趋势图。
3. User (1) <-> (N) DietLog：用户拥有多日的饮食日志，日志结构为 { "YYYY-MM-DD": { "breakfast": [...], "lunch": [...] } }。
4. Recipe (N) <-> (1) Admin：食谱由管理员统一创建 and 维护，用户可进行查阅 and 收藏。

4.3.2 数据库表结构设计（JSON 模式）
由于系统采用 JSON 存储，其结构设计如下：

1. 用户信息表 (users.json)
字段名 | 类型 | 说明
id | String | 唯一标识符（时间戳）
username | String | 登录用户名
password | String | Bcrypt 加密后的密码
role | Enum | 用户角色（admin/user）
profile | Object | 体征档案（含身高、体重、目标等）
weightHistory | Array | 体重历史（含 date, weight）
dietLogs | Object | 饮食日志嵌套对象（按日期 Key 索引）
avatar | String | 用户头像 Base64 字符串

2. 食谱信息表 (recipes.json)
字段名 | 类型 | 说明
id | Number | 食谱 ID
title | String | 食谱名称
calories | Number | 总热量（kcal）
carbs | Number | 碳水化合物（g）
protein | Number | 蛋白质（g）
fat | Number | 脂肪（g）
ingredients | Array | 原料列表
instructions | String | 烹饪步骤
image | String | 食谱展示图片 URL

第5章 系统实现

5.1 用户端设计与实现

5.1.1 仪表盘模块实现
仪表盘（Dashboard.vue）作为系统核心展示页，采用了响应式网格布局。
1. 热量统计组件：通过计算属性实时汇总当日饮食日志中的热量总和，结合 el-progress 圆形进度条展示与目标值的差距。进度条颜色根据达成度动态变化（绿色代表正常，红色代表超标）。
2. 计划槽位设计：将早、中、晚三餐设计为交互槽位。用户点击槽位后，可以查看已制订的计划或跳转至食谱库进行关联，实现了饮食记录与计划的统一管理。
3. 运动模式切换：集成了运动热量补偿逻辑。当用户开启“运动模式”并滑动滑块设定消耗热量后，系统会自动调高今日的总热量目标，并动态计算完成运动所需的参考时长（如慢跑、力量训练所需时间）。

5.1.2 饮食追踪与录入实现
1. 时间轴记录：饮食追踪页以垂直时间轴形式展示用户一日四餐。每项记录显示食物名称、热量及营养素分布。
2. 多模式录入：支持手动精确录入及收藏夹快速录入。核心亮点在于 AI 录入窗口，用户通过文本输入框提交描述后，前端展示 AI 脉冲加载动画，解析成功后以卡片形式展示解析出的各营养指标，确认后存入 dietStore 并同步至后端。

5.1.3 数据可视化实现
系统集成了 ECharts 可视化引擎。在仪表盘底部，通过折线面积图展示用户的体重变化趋势。由于 JSON 存储的特性，前端在获取 weightHistory 数组后，利用 map 函数提取日期（x轴）和体重数值（y轴），并设置 smooth: true 使曲线更加平滑，增强了数据反馈的直观性。

5.2 管理员服务端设计与实现

5.2.1 后台管理实现
管理员登录后，路由将自动导航至 AdminDashboard.vue。
1. 系统概览：通过顶部卡片展示系统当前的总用户数、总食谱数，数据通过后端 /api/admin/stats 接口汇总计算得出。
2. 数据监控：以列表形式展示注册用户的基本信息、角色及活跃状态，辅助管理员了解平台运营情况。
3. 食谱库 CRUD：管理员拥有对公共食谱库的完整控制权。通过 Element Plus 的表单组件，管理员可以录入包含图片 URL、详细配料和烹饪步骤的专业食谱，这些数据直接持久化到服务器端的 recipes.json。

5.3 系统特色功能

5.3.1 AI 生成一日三餐计划
系统利用大模型的生成能力，结合用户的个性化档案（性别、目标、忌口等）实现“一键制订”。后端 Prompt 明确要求 AI 兼顾营养配比与多样性。生成的计划不仅包含食物名称，还包含详细的预估热量和宏量元素指标。

5.3.2 AI 自然语言识别饮食
这是系统解决用户录入难的核心功能。系统后端采用了“双重校验”逻辑：首先清洗用户输入的干扰信息，然后利用 AI 的知识库估算常见餐食的热量。例如，当用户输入“一个汉堡和一杯可乐”时，AI 能自动拆解并按标准份量计算出约 900-1000kcal 的热量，极大提高了记录效率。

5.3.3 前端图像质量压缩
为解决 Base64 图片存储导致的 JSON 文件过大问题，系统在 compress.js 中实现了基于 Canvas 的压缩算法。该算法在上传前自动检查图片尺寸，按比例缩放至最大 800px 宽度，并以 0.7 的质量系数进行二次压缩。测试表明，该技术能将原始 2-5MB 的照片压缩至 100KB 左右，且视觉清晰度不受显著影响。
